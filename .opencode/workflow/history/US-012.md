## Description
**Como** Auditor de Seguridad, **quiero** generar informes de auditoría dentro de un proyecto **para** documentar hallazgos, no conformidades y recomendaciones.

### Acceptance Criteria
- AC-01: El informe incluye: alcance, criterios de auditoría, hallazgos, no conformidades, observaciones y recomendaciones.
- AC-02: Los informes se asocian a un proyecto de tipo AUDITORIA_ISO27001.
- AC-03: El Gerente General recibe notificación al completarse el informe.

### API Routes:
- `GET /api/informes-auditoria` — Listar informes (con filtros por proyecto, estado)
- `POST /api/informes-auditoria` — Crear informe
- `GET /api/informes-auditoria/[id]` — Obtener detalle
- `PUT /api/informes-auditoria/[id]` — Actualizar informe
- `DELETE /api/informes-auditoria/[id]` — Eliminar borrador

### Modelo InformeAuditoria:
- id (UUID, PK)
- proyecto_id (FK → Proyecto)
- creador_id (FK → Empleado)
- alcance (text)
- criterios_auditoria (text)
- hallazgos (JSON — array de objetos con titulo, descripcion, severidad, evidencia)
- no_conformidades (JSON — array de objetos con descripcion, clasificacion, accion_correctiva)
- observaciones (JSON — array de strings)
- recomendaciones (JSON — array de strings)
- estado (enum: BORRADOR, COMPLETADO)
- fecha_emision (date, nullable)
- created_at, updated_at

### Decisiones técnicas:
- El modelo `InformeAuditoria` usa campos JSON para hallazgos, no conformidades, observaciones y recomendaciones (flexibilidad)
- Solo proyectos de tipo AUDITORIA_ISO27001 pueden tener informes (validación RF-49)
- Solo AUDITOR, GERENTE_GENERAL y CISO pueden crear informes
- Al completar un informe, se notifica automáticamente al Gerente General (AC-03)
- Los informes completados no pueden modificarse ni eliminarse (inmutabilidad)
- Paginación y filtros por proyecto/estado en la lista
